# 人物検出・骨格推定・動作分類システム 実装仕様書

## 1. 目的

本仕様書は、映像入力から人物を認識し、人物ごとの動作をリアルタイムに分類するシステムの実装方針を定義するものである。

本システムは商用利用可能性の探索を目的とし、以下を重視する。

* 非スポーツ領域の日常・業務系動作の認識
* エッジ寄り環境での実行可能性
* リアルタイム性
* 将来的な独自データによる拡張性
* 人物検出・人数把握系機能との接続容易性

本システムの第一段階では、自然言語説明ではなく、あらかじめ定義した動作ラベルを人物ごとに出力する方式を採用する。

---

## 2. 採用方針

### 2.1 タスク定義

本システムの主タスクは以下とする。

1. 動画フレームから人物を検出する
2. 各人物に対して骨格を推定する
3. 同一人物を時系列に追跡する
4. 骨格時系列から人物ごとの動作ラベルを分類する
5. 必要に応じて人物数・通過人数・滞在人数などの上位機能へ接続する

### 2.2 採用アーキテクチャ

基本構成は以下とする。

**映像入力 → 人物検出/追跡 → 骨格推定 → 骨格時系列バッファ → 動作分類 → 出力**

第一候補の構成は以下とする。

* 骨格推定: **RTMO**
* 動作分類: **BlockGCN**

代替候補は以下とする。

* 骨格推定: RTMO
* 動作分類: CTR-GCN

### 2.3 採用理由

本用途では、人物単位の動作分類、人数把握との接続、背景変動への頑健性、計算量を重視するため、RGB-based 動画分類よりも skeleton-based 構成を主軸とする。

この方式の利点は以下である。

* 背景・照明・服装依存を相対的に抑えやすい
* 入力表現がコンパクトで後段分類器が軽量
* 人物ごとの動作分類に自然に接続できる
* 人数カウントや動線解析と親和性が高い
* 将来的な独自ラベル追加に向く

## 3. 対象ユースケース

### 3.1 想定ユースケース

* 店舗・施設内での人物行動把握
* 入退室・滞在・待機・接近などの行動判定
* 人物数カウントとの併用
* 人物ごとの状態ラベル付与

### 3.2 非対象

以下は第一段階の対象外とする。

* スポーツ特化の複雑動作解析
* 自然言語による長文説明生成
* 完全な意図推定や感情推定
* RGB物体情報に強く依存する行動の高精度認識

## 4. システム要件

### 4.1 入力要件

* 入力形式: USBカメラ、動画ファイル
* 解像度: 1280x720 または 1920x1080 を基本
* フレームレート: 15〜30 FPS
* カメラ位置: 固定カメラを基本
* 画角: 人体全身または上半身が安定して観測できること

### 4.2 出力要件

人物ごとに以下の情報を出力する。

* track_id
* bbox
* skeleton keypoints
* action_label
* action_score
* timestamp

必要に応じて以下も出力する。

* zone_id
* in/out event
* dwell_time
* aggregated counts

出力形式は JSON を基本とする。

#### 出力例

```json
{
  "timestamp": 1710000000.123,
  "persons": [
    {
      "track_id": 3,
      "bbox": [312, 145, 420, 398],
      "action_label": "standing",
      "action_score": 0.91,
      "keypoints": [[...]],
      "zone_id": "entrance"
    }
  ],
  "summary": {
    "num_persons": 2,
    "num_entered": 1,
    "num_exited": 0
  }
}
```

### 4.3 性能要件

初期目標は以下とする。

* 単一 GPU 環境での実時間処理
* 目標推論速度: **15 FPS 以上**
* 推奨目標: **20〜30 FPS**
* 同時人物数: 1〜5 人程度を主対象
* 動作分類遅延: 1〜2 秒以内

CPU only は参考実装に留め、商用PoCの本命は GPU 環境とする。

### 4.4 ハードウェア要件

#### 推奨

* GPU: RTX 5080 級
* VRAM: 16GB 以上推奨
* CPU: 8コア以上
* RAM: 32GB 以上

#### 最低実運用候補

* GPU: RTX 5060 級
* VRAM: 8GB
* CPU: 8コア以上
* RAM: 16GB 以上

#### 検証用

* CPU only 環境は機能確認用途に限定

## 5. モデル構成

### 5.1 前段: 人物検出・追跡

#### 目的

* フレーム内の人物領域を抽出する
* 同一人物を時系列で追跡する
* 骨格系列の人物対応を維持する

#### 要件

* 追跡IDの一貫性
* 一時遮蔽時の復帰
* 少人数環境での安定動作

#### 候補

* RTMO の人物・姿勢一体利用
* 必要に応じて軽量 detector + tracker を併用

### 5.2 骨格推定

#### 採用候補

* **RTMO**

#### 理由

* real-time multi-person pose estimation 向け
* one-stage で高速
* OpenMMLab 系で実装基盤が整っている

#### 入力

* RGB フレーム

#### 出力

* 人物ごとの 2D keypoints
* confidence
* bbox

### 5.3 動作分類

#### 第一候補

* **BlockGCN**

#### 第二候補

* **CTR-GCN**

#### 入力

* track_id ごとに蓄積した時系列 keypoints
* クリップ長 T フレーム分

#### 出力

* 人物ごとの動作ラベル
* 確信度

### 5.4 時系列窓設計

* 基本窓長: 16〜48 フレーム
* 初期値推奨: 24 または 32 フレーム
* 推論更新周期: 毎フレームまたは 2〜4 フレームごと
* 窓はスライディング方式

短すぎる窓は動作の文脈不足、長すぎる窓は遅延増加を招くため、初期実装では 1 秒前後を目安とする。

## 6. 認識ラベル設計

### 6.1 初期ラベル方針

第一段階では、骨格だけで識別しやすいラベルを優先採用する。

#### 初期候補ラベル

* standing
* sitting
* walking
* bending
* crouching
* lying
* hand_raising
* hand_waving
* approaching_zone
* leaving_zone
* handing_object
* falling

### 6.2 Skeleton-only で扱いやすいラベル

* 立つ
* 座る
* 歩く
* しゃがむ
* 前屈み
* 転倒
* 手を上げる
* 手を振る
* 接近
* 離脱

### 6.3 RGB補助が必要になりやすいラベル

以下は第一段階では主対象外、もしくは精度低下を許容して扱う。

* スマホ操作
* キーボード操作
* ボトル開閉
* 箱の開閉
* バッグへの物の出し入れ

### 6.4 ラベル設計上の注意

以下のような主観的ラベルは避ける。

* 迷っている
* 不審
* 急いでいる
* 考え込んでいる

これらは観測可能な行動に分解する。

## 7. データセット方針

### 7.1 初期段階

初期開発では、公開 skeleton action recognition データセットを用いてモデル選定と実装検証を行う。

候補:

* NTU RGB+D
* NTU RGB+D 120

目的:

* 基盤モデルの再現
* 推論パイプライン検証
* ラベル設計の初期検討

### 7.2 商用PoC段階

独自ユースケースに適合させるため、自前動画データセットの収集と fine-tuning を前提とする。

#### 収集方針

* 固定カメラ環境で撮影
* 実運用想定に近い背景・照明条件
* 各ラベルを十分なバリエーションで収集
* 負例・曖昧例を含める

#### アノテーション方針

* clip-level または person-level ラベル
* 必要に応じて開始・終了時刻付与
* ラベル定義書を別途作成

### 7.3 データ分割

* train / val / test を人物またはシーン単位で分離
* 同一シーン漏洩を避ける
* 実運用カメラごとの差異を考慮する

---

## 8. 推論パイプライン仕様

### 8.1 フレーム処理

1. フレーム取得
2. 人物検出または人物付き骨格推定
3. track_id 付与
4. keypoints を track_id ごとのバッファに格納
5. 規定窓長に達したら動作分類
6. ラベル平滑化
7. 描画・JSON出力

### 8.2 バッファ管理

* track_id ごとにリングバッファを保持
* バッファ長は T フレーム
* track 消失時は一定時間保持後に破棄
* 短時間欠損は補間または前値保持を許容

### 8.3 ラベル安定化

生のクラス出力はフレーム間で揺れやすいため、以下を導入する。

* 移動平均
* 多数決
* confidence threshold
* 最短継続時間制約

#### 例

* 3フレーム連続で同一クラス優勢なら確定
* confidence < threshold の場合は previous label を保持

## 9. ソフトウェア構成

### 9.1 使用候補スタック

* Python 3.10 以上
* PyTorch
* MMAction2
* MMPose
* OpenCV
* NumPy
* ONNX Runtime または TensorRT（将来最適化）

### 9.2 モジュール構成

#### `video_input/`

* カメラ入力
* 動画ファイル入力

#### `pose/`

* RTMO 推論ラッパ
* keypoint整形

#### `tracking/`

* track_id 管理
* 人物対応付け

#### `action/`

* BlockGCN 推論ラッパ
* クリップ生成
* ラベル安定化

#### `pipeline/`

* 推論全体制御
* 非同期処理

#### `output/`

* overlay 描画
* JSON 出力
* ログ保存

#### `config/`

* モデルパス
* 閾値
* 窓長
* クラス定義

### 9.3 設定ファイル例

```yaml
input:
  source: webcam
  width: 1280
  height: 720
  fps: 30

pose:
  model: rtmo-l
  conf_thr: 0.4

action:
  model: blockgcn
  clip_len: 32
  stride: 2
  score_thr: 0.6
  smooth_window: 5

tracking:
  max_missing_frames: 15

output:
  draw_overlay: true
  save_json: true
  save_video: false
```

## 12. リスクと対策

### 12.1 骨格欠損

#### リスク

* 遮蔽や視野外で keypoints が欠落する

#### 対策

* confidence ベースで欠損処理
* 補間
* track 消失猶予

### 12.2 ラベル揺れ

#### リスク

* standing / walking などが頻繁に切り替わる

#### 対策

* スムージング
* 継続時間制約
* class score threshold

### 12.3 物体依存動作の誤分類

#### リスク

* skeleton-only では phone, keyboard などが不安定

#### 対策

* 初期ラベルから除外
* 将来 RGB 補助を追加

### 12.4 ドメインギャップ

#### リスク

* 公開データで学習したモデルが実運用環境で劣化

#### 対策

* 早期に自前データ収集
* fine-tuning
* カメラ固定条件で評価
